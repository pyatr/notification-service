## Локальный запуск

1) Создать конфиг по умолчанию. В .env.example стоят все значения по умолчанию подходящие для локальной разработки.
```
cp .env.example .env
```

2) Запустить сборку через команду make. Это соберет контейнер сервиса и nginx.
```
make builddev
```

3) Поднять сервис. При успешном подтянии сервис будет доступен по адресу <a>http://127.0.0.1:8200</a>
```
make up
```

4) Запустить сидер. Это создаст пользователя для ручного тестирования. (Опционально) 
```
make seed
```

Makefile содержит и другие команды для быстрых действий, например для запуска анализов или тестов.

## Архитектура

- Notification Service - Laravel-сервис на основе php8.4-fpm-alpine
- Nginx - сервер для отправки HTTP-запросов к основному сервису
- Postgres - база данных
- RabbitMQ - брокер сообщений, используемый в сервисе для отправки оповещений пользователям

Отправка оповещений сделана через очереди для распределения нагрузки и возможности повторения при неудачной отправке.

## Улучшения для полноценной выкладки в продакшн

- Доработка docker-compose и supervisor.ini (локально используются docker-compose.dev.yml и supervisor.dev.ini).
- Команда в Laravel для ручного создания отдельной базы данных в Postgres, локально используется автоматически создаваемая при первом поднятии контейнера база данных. В эту команду передается название соединения, основной базы данных и логин с паролем от рутового пользователя от лица которого будет создаваться отдельная база данных для самого сервиса.
- Использование Dead Letter Exchange в RabbitMQ чтобы неудавшиеся работы не повторялись регулярно, но и не удалялись, а откладывались на потом.
- Слияние nginx и основного сервиса. Т.к. по умолчанию PHP сам не принимает HTTP-запросы, был добавлен контейнер с сервером для получения запросов извне и от других сервисов, что в некоторой степени усложняет конфигурацию и в теории создает лишнюю нагрузку чем если бы сервер находился в самом сервисе.
